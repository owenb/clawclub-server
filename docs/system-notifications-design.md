# System Notifications Design

## Problem

Today a club admin who spans multiple clubs has to fan out across `clubadmin.admissions.list` for each club to answer "any applications to approve?".

That happens because the current admission submission path only appends a club-scoped activity event:

- `admission.submitted` is written to `club_activity`
- admins can see it in `/updates`
- but it is only a timeline event, not a durable per-admin pending-work item

So the server has history, but not a real queue of "things this member still needs to know about".

We need a first-class system-notification surface that can carry:

- pending admin work: "a new application needs review"
- account state: "your subscription expires in 3 days"
- one-off delivery events: "we sent an email to x"

and surface all of it through the existing `/updates` transport.

## Load-Bearing Constraint

The current `/updates` model is cursor-forward.

That matters more than it first appears.

Today all three update sources are event-like:

- `club_activity`
- `signal_deliveries`
- `dm_inbox_entries`

They are read as "what is newer than this cursor?".

That is correct for timeline events. It is not correct for pending work.

Example:

1. Admin polls `/updates`
2. sees `notification.admission.submitted`
3. does nothing
4. polls again later

If the notification is treated like a cursor-forward signal, it disappears after first sight even though the admission is still pending.

That fails the actual requirement.

So this design problem is not just "where should rows be stored?". It is also "what read semantics should system notifications have?".

## Design Goal

Keep one `/updates` surface, but stop pretending every update source has the same semantics.

We need two distinct models:

- event stream semantics for activity, signals, and DM nudges
- queue semantics for system notifications

That means system notifications should be a first-class source in the updates read model, not just another signal topic stuffed into the existing cursor query.

## Recommendation

Introduce a new `notification` source in `/updates`.

Do not reuse `signal_deliveries` directly as the notification transport.

Instead:

1. Keep `signal_deliveries` for recommendation-style `signal.*` topics.
2. Add a first-class notification queue abstraction for `notification.*` topics.
3. Prefer compute-on-read for notifications that are fully derivable from current durable state.
4. Use a materialized notification table only for one-off events that are not derivable from current state.
5. Add `clubadmin.admissions.get` regardless of the notification design.

This is a cleaner split than forcing queue semantics into the current signal tape.

## Why Not Just Reuse `signal_deliveries`?

Reusing `signal_deliveries` is the smallest diff, but it is not the cleanest design.

Problems:

- the current signal read path is cursor-driven, so it does not naturally keep pending work visible
- the current signal schema and docs are recommendation-specific: `match_id`, `suppression_reason`, read-time entity filters, tuning-oriented acknowledgement semantics
- the signal query in `src/postgres.ts` is already topic-aware; adding operational notification logic there pushes it toward a god query

The repo docs already describe `signal_deliveries` as more general than synchronicity, but the actual behavior is still signal-shaped.

## Proposed Model

### 1. Add a Real Notification Source

Extend `PendingUpdate.source` with:

- `notification`

Meaning:

- `activity` = cursor-tracked club timeline
- `signal` = cursor-tracked targeted recommendations
- `inbox` = cursor-tracked DM nudges
- `notification` = current pending or unacknowledged system notices

The important part is semantic, not just naming:

- `notification` items are not hidden merely because the cursor moved past their creation time

## 2. Two Notification Backends

System notifications should have one read surface but two production paths.

### Derived notifications

These are computed from current state at read time.

Use for cases where the notification is just another view of durable business state.

Examples:

- admission still `submitted`
- membership or subscription expires soon
- payment is past due and still unresolved

Benefits:

- no write-time fanout
- no dedupe key
- no backfill when roles change
- no invalidation job when state changes

### Materialized notifications

These are stored rows for events that are not derivable later.

Use for cases like:

- "we sent an email to x"
- one-off moderation or operational notices
- explicit announcements generated by the system

These need durable stored rows because the event itself matters, not just current state.

## 3. Common Notification Shape

Whether a notification is derived or materialized, the read model should project the same shape into `/updates`.

Required fields:

- `updateId`
- `source = 'notification'`
- `recipientMemberId`
- `clubId` nullable
- `topic`
- `payload`
- `createdAt`

Recommendation:

- use stable notification keys
- expose `updateId` as `notification:<notificationKey>`

Examples:

- `notification:admission-submitted:<admissionId>:v1`
- `notification:subscription-expiring:<membershipId>:2026-04-12`
- `notification:email-sent:<deliveryId>`

That gives one uniform acknowledgement key even when some notifications are virtual and some are stored.

## 4. Separate Notification Receipts

Do not overload `signal_deliveries.acknowledged_state` for notifications.

Add a new receipt table, for example:

- `member_notification_receipts`

Keyed by:

- `recipient_member_id`
- `notification_key`

with:

- `state in ('processed', 'suppressed')`
- `suppression_reason`
- timestamps and versioning as needed

Why a separate receipt table is cleaner:

- it works for both derived and materialized notifications
- it avoids recommendation-specific fields leaking into operational notifications
- it lets `updates.acknowledge` route `notification:` items cleanly without touching signal storage

## 5. Polling And SSE Semantics

This is the part the first pass under-specified.

### Polling

`updates.list` should merge:

- cursor-forward event items from activity, signals, and inbox
- current active notification items irrespective of cursor

So on every poll:

- event sources are "what changed since cursor"
- notification source is "what is still pending for this member"

That is the queue behavior we actually need.

### SSE

SSE cannot just reuse the existing "while items.length > 0, continue" loop for sticky notifications or it will resend them forever.

Recommended behavior:

- on connection start, emit current notification items once
- keep a connection-local set of already-sent notification keys
- after wakeup, emit only newly-visible notification items or items whose key changed

Events remain cursor-driven. Notifications remain queue-driven.

This keeps polling and streaming aligned without turning the stream into an infinite replay loop.

## Admissions

Admissions are the motivating case, and they should be the first derived notification.

### Read model

For any actor who is a `clubadmin` in one or more clubs, compute:

- all current admissions in those clubs where status = `submitted`

and project each one into:

- `topic = 'notification.admission.submitted'`
- `updateId = notification:admission-submitted:<admissionId>:<submittedVersionNo>`

The notification disappears automatically when the admission stops being `submitted`.

### Why derived is better here

Derived admissions notifications avoid all of the ugly edge cases of materialized fanout:

- no backfill when someone is promoted to clubadmin
- no dedupe coordination between write path and backfill path
- no read-time invalidation against a materialized row
- no orphan rows for neglected admissions

For this use case, the admission itself is the source of truth.

## Notification Payload Design

Keep notification payloads refs-first.

For admissions, prefer:

- `admissionId`
- `clubId`

Optional:

- small display fields if product UX truly needs them

But the authoritative follow-up should be:

- `clubadmin.admissions.get`

That keeps notification payloads from drifting into denormalized mini-records.

## `clubadmin.admissions.get`

This action should exist regardless of notification design.

Recommended input:

- `clubId`
- `admissionId`

Recommended output:

- `admission: AdmissionSummary`

Why it matters:

- notifications should point to one concrete object
- agents should fetch that object directly
- forcing a list-and-filter path is unnecessary friction

This is a good standalone change even before notification work starts.

## Materialized Notification Table

For non-derivable notifications, add a dedicated table, for example:

- `member_notifications`

Suggested fields:

- `id`
- `notification_key`
- `recipient_member_id`
- `club_id` nullable
- `topic`
- `payload`
- `dedupe_key` nullable
- `created_at`
- optional `expires_at` or `invalidated_at`

Notes:

- `club_id` should be nullable to support account-global notifications
- materialized notifications should wake the existing `updates` notifier on insert
- dedupe should be generic, not recommendation-specific

## Global Notifications

Some notifications will not belong to any club.

Examples:

- account emails
- platform-level notices

So notification scope must allow:

- `clubId = null`

This is simpler in a dedicated notification source than in the current signal query.

Also note:

- `/updates` does not currently expose a user-supplied club filter

So showing global notifications alongside club-scoped ones is not an API ambiguity in the current transport.

## Cleanup And Growth

Derived notifications do not accumulate storage.

Materialized notifications do, so the design should include a cleanup answer from the start.

Reasonable default:

- retain acknowledged or invalidated materialized notifications for a bounded period, then sweep

Example:

- delete or archive old rows after 90 days

That does not need to block phase 1, but it should be explicit.

## Concrete Implementation Slices

### Phase 1

1. Add `clubadmin.admissions.get`
2. Extend `PendingUpdate.source` with `notification`
3. Add repository support for listing derived admission notifications
4. Merge notification items into `updates.list` irrespective of cursor
5. Update SSE loop so notifications are emitted once per connection, not endlessly

This alone solves "any applications to approve?" cleanly.

### Phase 2

1. Add `member_notifications` for materialized event notices
2. Add `member_notification_receipts`
3. Extend `updates.acknowledge` to support `notification:` IDs
4. Add first event-backed notification, such as `notification.email.sent`

### Phase 3

1. Add more derived notifications where state already exists
2. Add more materialized notifications where the event itself must be preserved
3. Add retention sweep for old materialized notifications

## What Not To Do

### Do not solve this with a cross-club "check all admissions" action

That solves only one question and creates another parallel access path instead of a reusable notification system.

### Do not use `club_activity` as the queue

`club_activity` is history, not pending work.

### Do not force notification semantics into the signal tape

Signals and notifications may share transport infrastructure, but they do not share the same invariants.

The clean design is to acknowledge that directly in the read model.
